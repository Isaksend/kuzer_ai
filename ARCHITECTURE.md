# Архитектура и технические решения (Zone Monitor)

В этом документе описаны ключевые архитектурные решения, особенности реализации и причины выбора тех или иных библиотек для проекта Zone Monitor.

## 1. Выбор стека технологий

### YOLOv8 (от Ultralytics)
**Почему:**
* На данный момент это state-of-the-art модель для object detection в реальном времени.
* Из коробки поддерживает встроенный трекинг (BoT-SORT / ByteTrack), что избавляет от необходимости писать отдельный алгоритм сопоставления ID объектов между кадрами.
* Крайне прост в инференсе (буквально одна строчка `model.track()`).
* Модель `yolov8n.pt` (nano) работает с приемлемым FPS даже на слабых CPU без дискретной видеокарты.

### OpenCV (cv2)
**Почему:**
* Индустриальный стандарт для работы с компьютерным зрением и видеопотоками.
* Быстрые функции для отрисовки графики поверх кадра (`cv2.rectangle`, `cv2.polylines`, `cv2.addWeighted`).
* Поддержка чтения различных источников: локальные файлы, IP-камеры, RTSP-потоки (`cv2.VideoCapture`).
* Быстрая математика проверки вхождения точки в полигон: функция `cv2.pointPolygonTest` сильно экономит ресурсы процессора.

### FastAPI + Uvicorn
**Почему:**
* Отличная замена Flask для современных приложений.
* Нативная поддержка асинхронности (`async`/`await`), что необходимо для работы с WebSocket.
* Очень быстрый, легко настраиваемый (из коробки есть монтирование статики и CORS).
* Полноценная поддержка REST API с автоматической документацией.

### WebSockets
**Почему:**
* Обычный HTTP long-polling или MJPEG не подходят для современного дашборда с мгновенной отзывчивостью.
* WebSocket держит одно TCP-соединение и позволяет отправлять JPEGs-блоки напрямую в браузер без оверхеда HTTP-заголовков на каждый кадр.
* В браузере (через `Blob` и `URL.createObjectURL(blob)`) это отображается с нулевой задержкой и не нагружает DOM-дерево.

## 2. Особенности реализации

### Многопоточный захват видео (VideoStream)
Обычно чтение через `cap.read()` блокирует поток выполнения. Если модель YOLO делает inference 50мс, а затем `cap.read()` ждёт кадр по сети 30мс, итоговый FPS сильно падает.
* **Решение:** Слой `VideoStream` (в файле `video_stream.py`). В нём создаётся отдельный демон-поток (`threading.Thread`), который непрерывно читает последний доступный кадр из RTSP/камеры с максимальной скоростью и кладет его в переменную. YOLO берет просто последний актуальный кадр (`bufferless` чтение для RTSP), что полностью исключает буферизацию (задержку сигнала в несколько секунд), которая является частой проблемой OpenCV.
* **Троттлинг для фалов:** Для обычных `.mp4` файлов, наоборот, чтение происходит слишком быстро, поэтому в поток добавлено ручное время ожидания (дросселирование), чтобы видео воспроизводилось со скоростью реального времени (1x FPS).

### Кольцевой буфер и «Черный ящик» (VideoRecorder)
* **Решение:** Вместо того чтобы начинать записывать видео *только с момента* нарушения, `VideoStream` держит в памяти кольцевой буфер (`collections.deque`) на последние 10 секунд (250 кадров при 25 FPS).
* Когда срабатывает ALARM, вызывается фоновый поток `VideoRecorder`, который склеивает кадры из буфера (ДО) с последующими 10 секундами живого потока (ПОСЛЕ), позволяя службе безопасности увидеть контекст: откуда появился нарушитель.
* В качестве fallback-механизма кодека мы пробуем `XVID`, `MJPG` или `H264` (разные Windows-системы имеют разные пакеты предустановленных кодеков).

### Hot-Reload Зон на лету
* Чтобы избежать необходимости перезапускать весь backend-сервер каждый раз, когда меняются зоны в редакторе, используется событийная модель через `threading.Event` (`zones_changed`).
* Пользователь рисует зоны в Canvas (браузер) → отправляет POST-запрос на `/api/zones`.
* API сохраняет зоны в `restricted_zones.json` (per-video, по ключу источника) и "дергает" `Event`.
* Рабочий поток (worker), который крутит YOLO, на следующей итерации цикла видит этот флаг, перечитывает JSON с диска и применяет новые зоны без потери контекста трекера.

### Подсчет времени в зоне (Zone Durations)
Нельзя просто замерять "вход" и выход, так как человек может спрятаться за препятствием (трекер потеряет ID) или мерцать на границе зоны.
* **Структура:** Словарь `{ id: {"enter": timestamp, "total": float} }`
* Каждый кадр алгоритм проверяет координаты ног (foot = нижняя центральная точка Bounding Box). Если `foot` внутри полигона, проверяется, есть ли у `id` время `enter`. Если объекта в текущем кадре нет, или он вышел за зону, прошедшее с `enter` время прибавляется к `total`, а `enter` сбрасывается (`None`). Таким образом аккумулируется только чистое время нахождения в кадре внутри запрещенной зоны.

### Telegram и Cooldown (TelegramNotifier)
Отправка фото в Telegram API (`requests.post`) занимает ~300-800 мс. Если ставить вызов внутри основного цикла YOLO, видеопоток "замерзнет" на секунду.
* **Решение:** Отправка вынесена в отдельный daemon-поток.
* **Cooldown механизм:** Чтобы не заспамить бота сотней фотографий одного и того же курьера, используется словарь блокировок: `self.last_sent[obj_id]`. Пока не пройдет 10 минут, второе фото этого же человека (`obj_id`) отправлено не будет.

## 3. Масштабируемость и дальнейшее развитие
* **База Данных:** Сейчас используется `log.csv` для простоты и переносимости. Для production-нагрузки потребуется миграция на SQLite или PostgreSQL (особенно для модуля Analytics).
* **Мониторинг нескольких источников:** Текущая архитектура позволяет запустить ровно 1 worker на дашборд. В будущем можно перенести worker в пул процессов (Multiprocessing или Celery), чтобы дашборд мог отрисовывать сразу 4-8-16 камер на одном экране.
* **GPU Инференс:** Так как используется стандартная Pytorch/Ultralytics обертка, перенос на GPU (CUDA, TensorRT) бесшовен — достаточно просто установить `torch` с поддержкой CUDA на машину с видеокартой NVIDIA, код менять не нужно.
